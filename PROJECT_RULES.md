# PROJECT RULES: Clean Architecture Refactoring (V-027)

> **상태**: ACTIVE
> **강제성**: ABSOLUTE (위반 시 작업 중단 및 사용자 보고)

## 1. 교차 검증 원칙 (The "Rule of Three")
모든 코드 변경 및 설계 결정은 **작성자 외 2개의 AI**가 반드시 검증해야 한다.
- **작성자**: 설계를 제안하거나 코드를 작성한 AI.
- **검증자 1 (Reviewer)**: 코드의 논리, 아키텍처 적합성, 잠재적 버그를 검토.
- **검증자 2 (Auditor)**: 검증자 1의 리뷰가 유효한지, **Fact-Check**가 되었는지 확인.
- **기록 필수**: 검증 주체와 결과(`PASS`/`FAIL`)를 `REFACTORING_PLAN.md`에 타임스탬프와 함께 기록해야 함.

## 2. 정직과 팩트 (Zero Hallucination Policy)
- **추측 금지**: 모르면 "모른다"고 하고 확인 절차를 거친다.
- **가설 금지**: 실행 결과나 공식 문서 없이 "될 것이다"라고 가정하지 않는다.
- **회피/대충 금지**: 에러 발생 시 근본 원인을 찾을 때까지 파고든다.
- **거짓 금지**: 특히 진행 상황에 대해 1%의 과장도 허용하지 않는다.
- **기반 정보**: 모든 판단은 실행 로그, 공식 문서, 검증된 소스 코드에만 기반한다.

## 3. 작업 기록 (Documentation)
- 모든 진행 상황(Phase, Step, Validation)은 실시간으로 파일(`REFACTORING_PLAN.md`)에 기록한다.
- 구두 보고(채팅)보다 파일 기록이 우선이다.

## 4. 버전 관리 (No Commit yet)
- Git Commit은 사용자의 명시적 지시 전까지 수행하지 않는다.
- 파일 시스템 변경(`write_to_file`)으로만 상태를 유지한다.

## 5. 실행 감시 (Mutual Surveillance)
- 작업을 수행하는 AI는 다른 AI(Reviewer)가 위 규칙을 지키는지 감시해야 한다.
- 위반 징후 포착 시 즉시 중단하고 사용자에게 이실직고한다.
